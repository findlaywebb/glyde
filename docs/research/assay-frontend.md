# Assay frontend ‚Äî patterns to reuse in Glyde

Mining notes from `/Users/findlaywebb/assay/frontend` (read-only). Assay is a SvelteKit +
FastAPI **blind-labelling** tool served same-origin over a home LAN to a phone ‚Äî almost
exactly Glyde's deployment shape (local-first, mobile, LAN, dyslexia/reading-adjacent UX
discipline). Glyde was already scaffolded from these patterns: the typed seam, dark-mode-first
tokens, boundaries config, and `ui/Button.svelte` are **already present and near-identical**.
This doc therefore captures the *mechanism exactly* where Glyde mirrors it, and concentrates on
the **net-new** pieces an RSVP / guided-sweep reader needs: the `.svelte.ts` reactive
state-machine factory, the gesture binding, the timing/animation approach, PWA shell, and the
LAN/iOS gotchas.

Versions are from `frontend/package.json` unless noted. Assay uses **bun**; Glyde uses **npm** ‚Äî
the only consequential difference is the `bun run` vs `npm run` script prefix.

## 1. Project structure & routing

```
src/
  app.html            # <html class="dark"> shell, viewport flags
  app.css             # Tailwind v4 @theme tokens (the whole palette)
  app.d.ts
  hooks.server.ts     # the single front door: proxy /api/* ‚Üí FastAPI, serve the rest
  lib/
    api/
      client.ts       # openapi-fetch client, baseUrl '/api'
      schema.d.ts     # GENERATED by openapi-typescript, committed
    components/        # element "ui": presentational, primitive props only
      ui/              # owned shadcn-svelte primitives (Button, Badge, Progress)
      LabelCard.svelte  VerdictBar.svelte  SessionProgress.svelte ...
    domains/           # element "domains": .svelte.ts state machines + pure logic
      labelling.svelte.ts   swipe.svelte.ts   swipe.ts
    utils.ts           # element "utils": cn() only
  routes/
    +layout.svelte     # app.css import, SW registration, <InstallHint/>
    +page.ts / +page.svelte           # session picker (list)
    sessions/[session_id]/
      +page.ts          # the load (typed-seam reads)
      +page.svelte      # thin: renders <LabellingLoop/>
      LabellingLoop.svelte  # route-local orchestration component
static/
  manifest.json  sw.js  icons/  robots.txt
```

The **five-element architecture** (`routes, domains, ui, api, utils`) is enforced by
`eslint-plugin-boundaries` v6 + `dependency-cruiser`. Edges point inward:
`routes‚Üí{domains,ui,api}`, `domains‚Üí{ui,api,utils}`, `ui‚Üí{utils}`; `api`/`utils` are sinks.
Glyde already has this exact config in `frontend/eslint.config.js`. Key subtlety from assay's
CLAUDE.md: **eslint-boundaries is the authoritative edge guard** (it resolves `$lib` via
`eslint-import-resolver-typescript`); **dependency-cruiser does NOT resolve `$lib`**, so its
element forbids only catch *relative* imports ‚Äî treat depcruise as cycle-detection + a
relative-import backstop, not the aliased-edge guard.

Routing notes worth copying:
- **Thin route component delegates to a local orchestration component.** `+page.svelte` is
  trivial; `LabellingLoop.svelte` (co-located, not in `$lib`) wires the state machine to the
  DOM. For Glyde: `reader/[doc_id]/+page.svelte` ‚Üí `ReaderLoop.svelte`.
- **`{#key}` to force a clean remount per item.** `{#key loop.card.item_id}<LabelCard .../>{/key}`
  resets per-card local state on advance. The RSVP analogue: key the word/segment surface on the
  segment id so transitions restart cleanly.
- **Keyed component for per-entity state.** `LabellingLoop` is keyed on `session_id` (remounts
  per session), so capturing `data.firstCard` once in a closure is correct.

## 2. The typed API‚Üîfrontend seam (Glyde's "typed seam") ‚Äî assay's mechanism, exactly

This is the load-bearing pattern. Reproduce it verbatim.

**Generation.** `openapi-typescript` (pinned `7.13.0`) generates `src/lib/api/schema.d.ts`
from the committed `docs/schemas/openapi.json`. Two scripts, both gates:
```json
"gen:api":       "openapi-typescript ../docs/schemas/openapi.json -o src/lib/api/schema.d.ts",
"check:api-drift":"openapi-typescript ../docs/schemas/openapi.json -o src/lib/api/schema.d.ts --check"
```
The generated file IS committed; `--check` fails CI if it drifts. A second backend gate proves
the committed `openapi.json` matches the live app ‚Äî **neither gate alone proves freshness**
(assay's CLAUDE.md is explicit about needing both).

**The client** (`src/lib/api/client.ts`) ‚Äî a one-liner over `openapi-fetch` (pinned `0.17.0`):
```ts
import createClient from 'openapi-fetch';
import type { paths } from './schema';
export const api = createClient<paths>({ baseUrl: '/api' });
```
A **module-level stateless client**. The base `/api` is what the *browser* uses (resolved
against `window.location.origin`).

**Consuming types in app code** ‚Äî import the generated `components['schemas'][...]`, never
hand-write:
```ts
import type { components } from '$lib/api/schema';
type ItemCard = components['schemas']['ItemCard'];
type ProgressView = components['schemas']['ProgressView'];
```

**Calling inside `load`** (`routes/.../+page.ts`) ‚Äî three rules: per-request `fetch`, an
absolute same-origin `baseUrl`, branch `{ data, error }`:
```ts
export const load: PageLoad = async ({ fetch, params, url }) => {
  // SSR needs an ABSOLUTE base: openapi-fetch builds new Request(base+path) BEFORE the
  // provided fetch runs, and a relative URL has no origin in Node. event.fetch re-enters
  // the proxy in-process ‚Üí same-origin, no CORS.
  const baseUrl = `${url.origin}/api`;
  const [headerRes, cardRes] = await Promise.all([        // independent reads in parallel
    api.GET('/sessions/{session_id}', { fetch, params: { path }, baseUrl }),
    api.GET('/sessions/{session_id}/next', { fetch, params: { path }, baseUrl })
  ]);
  if (headerRes.error) error(502, 'Could not load the session.');  // a 200 is NOT success
  if (cardRes.error)   error(502, 'Could not load the first card.');
  return { header: headerRes.data, firstCard: cardRes.data.card };
};
```

**Client-side mutations** use the *module* client (global fetch) via a small **gateway
interface** the route binds and injects into the state machine (keeps the machine
transport-agnostic and trivially mockable in tests):
```ts
const gateway: LabelGateway = {
  postAction: (action) => api.POST('/actions', { body: action }),
  nextCard:   () => api.GET('/sessions/{session_id}/next', { params: { path: { session_id } } }),
};
```

**Same-origin serving** (`src/hooks.server.ts`) ‚Äî the single front door, ~30 lines. `handle()`
early-returns for `/api*` (strip `/api`, reverse-proxy to `GLYDE_API_ORIGIN`/`ASSAY_API_ORIGIN`),
otherwise `resolve()`. Glyde already has this. Two non-obvious bits to keep:
- **Origin-assertion CSRF guard**: a non-GET with a *present* `Origin` that fails strict
  same-origin ‚Üí `403 {code:'forbidden_origin'}` *before* the upstream fetch. An *absent* Origin
  is allowed (a cross-origin browser POST always carries Origin per the Fetch spec, so
  absent ‚áí trusted local agent or in-process SSR re-entry).
- **`filterSerializedResponseHeaders`** must allow `content-type` + `content-length`, else a
  hydrating typed-seam read throws (SvelteKit strips replayed headers by default and
  `openapi-fetch` reads those two).

> For Glyde the seam is identical; the only deltas are the env var (`GLYDE_API_ORIGIN`) and the
> Digest IR types replacing the labelling schemas.

## 3. Svelte 5 runes idioms ‚Äî done well

Project-wide runes mode is forced in `svelte.config.js` (`compilerOptions: { runes: true }`),
with a `dynamicCompileOptions` escape so third-party `node_modules` `.svelte` files that still
use `export let` (e.g. the Storybook CSF addon) compile under legacy runtime:
```js
vitePlugin: {
  dynamicCompileOptions: ({ filename }) =>
    filename.includes('/node_modules/') ? { runes: undefined } : {}
}
```

**`$props` with a typed interface, defaults destructured, `...rest` spread** (every component):
```ts
interface Props extends HTMLButtonAttributes { variant?: Variant; size?: Size; children: Snippet; }
let { variant = 'default', size = 'default', type = 'button', class: className,
      children, ...rest }: Props = $props();
```

**`$derived` for anything that is a pure function of state ‚Äî never `$effect`.** This is *the*
discipline. Examples:
```ts
const percent  = $derived(max > 0 ? Math.min(100, Math.max(0, (value / max) * 100)) : 0);
const hasContext = $derived(context != null && context !== '');
const reviewItems = $derived(loop.labelledCards.map((l) => ({ ... })));  // mapping a list
```

**`$effect` only for genuine side effects, always with teardown.** Assay's effects are: SW
registration, imperative DOM listeners, focus management, and the gesture binding ‚Äî never
"compute state from state". Pattern for a DOM side effect:
```ts
$effect(() => {
  const el = cardRegion;
  if (!el) return;
  return bindSwipe(el, { onPass: () => loop.submit('pass'), ... });  // returns teardown
});
```
Focus-follow effect reads a value purely to subscribe to it:
```ts
$effect(() => {
  void loop.card?.item_id;                       // subscribe: re-run when the card swaps
  if (loop.status === 'labelling') cardRegion?.focus();
});
```

**Never destructure `$state`** (loses reactivity) ‚Äî the state machine returns **getters**, not a
plain object (see ¬ß4).

**`$props.id()`** for SSR-stable, per-instance unique ids (assay uses it for ARIA
`aria-controls`/`id` on disclosure panels when many cards mount at once):
```ts
const uid = $props.id();   // `${uid}-rubric`, `${uid}-context`
```

**Snippets (`{#snippet}` / `{@render}`)** replace slots. `children: Snippet` prop + `{@render
children()}`. In-file snippets for *local* repetition (assay's `LabelCard` shares one
`disclosure` snippet for its two collapsible panels ‚Äî rule-of-three: two uses ‚áí a snippet, not a
component).

**Props-down / callbacks-up, no `createEventDispatcher`.** `VerdictBar` takes `onPass`,
`onFail`, ‚Ä¶ callbacks; owns no state. `$bindable` only on genuine co-ownership.

## 4. The `.svelte.ts` reactive state-machine factory ‚Äî the key RSVP pattern

`src/lib/domains/labelling.svelte.ts` (`createLabelLoop`) is the single most reusable structure
for Glyde's reader. It is a **factory returning `$state`-backed getters + imperative methods** ‚Äî
a state machine that lives outside any component, is unit-testable, and is driven by the
keyboard/swipe/buttons identically.

Shape:
```ts
export function createLabelLoop(deps: LabelLoopDeps) {
  let status = $state<'labelling' | 'halted' | 'complete'>(...);
  let card   = $state<ItemCard | null>(initialCard);
  let busy   = $state(false);
  let progress = $state<ProgressView>(initialProgress);
  // ... imperative methods: submit(), undo(), end() ...
  return {
    get status() { return status; },     // getters, NOT a destructurable object ‚Äî preserves reactivity
    get card()   { return card; },
    get busy()   { return busy; },
    submit, undo, end, /* ... */
  };
}
```
Why getters: a component reads `loop.status` and stays reactive; returning `{ status }` would
snapshot the value and break.

Patterns to lift directly into an RSVP/sweep controller:
- **Dependency injection of the transport + an id source** (`gateway`, `newId?: () => string`)
  so tests are deterministic and the machine is DOM-free.
- **`busy` flag makes a cycle exclusive** ‚Äî no overlapping advances. For RSVP: a `playing`/
  `paused` flag plus a tick guard.
- **Optimistic-then-persist, imperative** ‚Äî mutate state synchronously (bump counts, announce),
  `await tick()` so the optimistic paint lands, *then* the network call. "Never effect-driven
  state."
- **An undo stack of in-flight promises** so a back-step chains onto the action it reverses.
- **A `halt = freeze` terminal state**: a rejected action sets `status:'halted'` + an error
  banner and accepts nothing further until reload. Glyde's reader wants the same for a failed
  Digest fetch mid-stream.
- **A pure key‚Üíintent map** isolated as `keyToIntent(key): Command | null` (a `switch`), so the
  keyboard handler in markup stays tiny. RSVP analogue: `Space`‚Üítoggle, `‚Üê/‚Üí`‚Üíprev/next word,
  `‚Üë/‚Üì`‚Üíspeed.

The factory holds **no `$effect`**; the window keydown listener lives in the route markup
(`<svelte:window onkeydown={handleKeydown}/>`). Keep effects at the DOM boundary, logic in the
machine.

## 5. Timing & animation ‚Äî what assay actually has (and the honest gap for RSVP)

**Important, verified:** assay has **no `requestAnimationFrame`, no `setInterval`, no
`performance.now` timing loop**. A grep of the whole frontend source finds exactly one timing
primitive: a single `setTimeout` in `swipe.svelte.ts` for the card fly-off settle. So assay
gives Glyde the *gesture / transition* toolkit, **not** a ready-made word-flashing clock ‚Äî Glyde
must build the RSVP scheduler itself (a `requestAnimationFrame` accumulator against
`performance.now`, or a self-correcting `setTimeout`). What assay *does* give:

**Compositor-only transforms for smooth motion** (`swipe.svelte.ts`) ‚Äî never touch layout on the
hot path. The drag follow writes only `transform`:
```ts
function follow(dx: number): void {
  const rot = Math.max(-MAX, Math.min(MAX, dx * ROTATION_PER_PX));
  target.style.transform = `translateX(${dx}px) rotate(${rot}deg)`;   // transform only ‚Üí GPU
}
```
For Glyde's **moving sweep marker / fade trail**: animate `transform: translateX()` and
`opacity` (both compositor-friendly), never `left`/`width`/`top`.

**CSS transitions for snap/settle** rather than JS animation where the motion is A‚ÜíB:
```ts
target.style.transition = `transform ${SETTLE_MS}ms ease-out`;   // SETTLE_MS = 200
target.style.transform  = 'translateX(0px) rotate(0deg)';        // snap-back
```
The progress bar fill is a pure CSS width transition: `transition-[width] duration-200` with
`style="width: {percent}%"` (`ui/Progress.svelte`). A guided-sweep marker can be a CSS
transition between word positions for the in-between glide.

**`prefers-reduced-motion` read ONCE at bind time**, then branch every animation on it ‚Äî a
reduced-motion user gets the instant outcome, no travel:
```ts
const animate = !prefersReducedMotion();   // window.matchMedia('(prefers-reduced-motion: reduce)')
function snapBack(): void { if (!animate) return resetInstant(); /* else transition */ }
```
This is **mandatory** for an RSVP reader (flashing/motion is an accessibility and seizure
concern) ‚Äî gate the word-flash cadence and any trail/marker motion on it.

**Settle-timer hygiene**: the `setTimeout` handle is tracked and `clearTimeout`'d on every new
gesture and on teardown, so a stale timer can't fire after unmount. The RSVP loop needs the same
`cancelAnimationFrame` on pause/teardown.

## 6. Gesture binding ‚Äî pointer events, axis-lock, DOM-isolated (reusable for tap/swipe controls)

`src/lib/domains/swipe.svelte.ts` (`bindSwipe(target, opts): () => void`) is a clean,
framework-agnostic input adapter ‚Äî directly reusable for Glyde's tap-to-pause / swipe-to-seek.

- **Pointer Events only** ‚Äî one model for touch/pen/mouse (`pointerdown/move/up/cancel`). No
  separate touch + mouse handlers.
- **Axis-lock on first move past `LOCK_THRESHOLD_PX` (8px):** horizontal ‚áí swipe (follow + commit
  or snap-back), vertical ‚áí yield entirely to native scroll and never transform. This is what
  stops a swipe fighting page/evidence scroll ‚Äî critical on a phone reading surface.
- **`setPointerCapture` feature-detected** (jsdom lacks it; wrapped in `try/catch`, non-fatal).
- **An "ignore zone"**: a gesture starting inside `[data-evidence-scroll]` is left to native
  scroll. Glyde's analogue: don't let a swipe on the reading text body trigger a seek if the user
  is selecting/scrolling.
- **Single primary pointer** guarded by `pointerId` so a second finger can't hijack.
- **Returned teardown** detaches every listener + clears the settle timer ‚Äî consumed by a
  `$effect` that returns it.

**The decision is split into a pure, DOM-free classifier** (`swipe.ts`, no `.svelte`), so the
*logic* is node-testable without a browser:
```ts
export function classifySwipe(t: SwipeTrajectory): 'pass' | 'fail' | null { ... }
```
Commit rule worth stealing for any threshold gesture: **distance OR velocity, not both** ‚Äî
`committedByDistance = distance >= width*0.45` OR `committedByFlick = distance >= 40 &&
velocity >= 0.3 px/ms`. Requiring both made the original "hard to hit the right speed".

This **pure-core / thin-DOM-shell split** is the testing-friendly shape for Glyde's RSVP
scheduler too: keep the cadence math (WPM ‚Üí ms/word, ORP offset, pause-at-punctuation) in a pure
`.ts`, and the `requestAnimationFrame` wiring in a thin `.svelte.ts`.

## 7. Component conventions

- **Typed `interface Props` + `$props()`**, defaults in the destructure, `...rest` spread onto
  the root element; `extends HTMLButtonAttributes`/`HTMLAttributes<HTMLDivElement>` to inherit
  native attrs.
- **Owned shadcn primitives** in `ui/` (`Button`, `Badge`, `Progress`) ‚Äî token classes only, no
  raw palette, `cn()` to merge:
  ```ts
  import { type ClassValue, clsx } from 'clsx';
  import { twMerge } from 'tailwind-merge';
  export function cn(...inputs: ClassValue[]): string { return twMerge(clsx(inputs)); }
  ```
  Variant/size maps as `Record<Variant, string>` of token classes (`Button.svelte`).
- **Presence-driven rendering, not config flags**: `LabelCard` renders evidence/context/rubric
  based purely on whether each string is non-empty (`hasRubric = $derived(rubric != null && rubric
  !== '')`) ‚Äî no per-type layout hint. Glyde's reader chrome can render optional Digest fields the
  same way.
- **`ui` components take primitive props, never the API type** (the boundary forbids `ui‚Üíapi`).
  The route flattens `data` to primitives before passing down. This keeps presentational
  components blind to the wire shape ‚Äî good for Glyde where the Digest IR will churn.
- **Module + component docstrings carry the "why"** ‚Äî assay's components open with a comment
  block stating purpose, invariants, and what they deliberately do *not* do. Matches Glyde's
  "docstring as agent contract" rule.

## 8. Styling / theming / tokens

- **Tailwind v4, `@theme inline`, semantic tokens only.** `app.css` defines the full palette as
  CSS custom properties on `:root` (light) and `.dark` (dark) in **oklch**, then exposes them to
  Tailwind via `@theme inline` so `bg-card` etc. re-resolve at runtime and flip under `.dark`.
  **`inline` is load-bearing** ‚Äî omit it and Tailwind snapshots the `:root` value at build time
  and dark mode silently breaks. (Glyde's `app.css` already has this verbatim.)
- **Dark-mode-first**: `<html lang="en" class="dark">` in `app.html` so the app paints dark on
  first render with no JS and no flash; `color-scheme` follows the cascade (`:root` light /
  `.dark` dark) so native UA surfaces (scrollbars, form controls, the iOS rubber-band overscroll
  area, the pre-paint root bg) match. `<meta name="theme-color" content="#0a0a0a">` in the
  layout. No in-app theme switcher in assay (Storybook toolbar only). For Glyde's dyslexia
  audience you will likely *add* an in-app theme/contrast/font control ‚Äî but the token cascade is
  the right substrate for it.
- **Custom dark variant**: `@custom-variant dark (&:is(.dark *));` (class-based, not media).
- **Accessibility-driven token tuning**: assay darkened `--destructive` in dark mode to clear
  WCAG AA on white text (the axe gate caught the original) ‚Äî palette values are an a11y concern,
  not just taste. Relevant to Glyde's reader-contrast requirements.
- A repeated arbitrary `[...]` value is treated as a **missing token**.

## 9. Vite / build config

- `vite.config.ts`: `plugins: [tailwindcss(), sveltekit()]` + a **dev proxy mirroring the prod
  hooks front door** so the browser is same-origin in dev too (no CORS in dev):

  const apiOrigin = process.env.ASSAY_API_ORIGIN ?? 'http://127.0.0.1:8000';
  server: { proxy: { '^/api(/|$)': { target: apiOrigin, changeOrigin: true,

  Anchored regex `^/api(/|$)` matches `/api` and `/api/‚Ä¶` but not `/apiary`.

## 10. PWA / LAN / iOS ‚Äî directly applicable to Glyde-on-a-phone

Assay and Glyde share the deployment: SvelteKit `adapter-node` server is the single LAN front
door, hit from a phone over plain HTTP. The mobile/PWA work transfers wholesale.

- **`static/manifest.json`**: `display: standalone`, `orientation: portrait`, dark
  `theme_color`/`background_color` (`#0a0a0a`), 192/512 + maskable icons, `start_url`/`scope`
  `/`. Linked from the layout with `<link rel="manifest">` + `apple-touch-icon`.
- **`static/sw.js` ‚Äî a shell-cache service worker, classic JS.** The one load-bearing property:
  **genuine passthrough for `/api/*`** (the `/api` check is the first statement in the fetch
  handler and returns `undefined` ‚áí no `respondWith` ‚áí straight to network), so a cache-miss can
  never serve a stale read. Caches only `/_app/immutable/**` (content-hashed) cache-first; never
  caches HTML navigations (SSR must reflect live state), cross-origin, or non-GET. The routing
  decision is a pure `handleFetch(event)` so it is node-testable (`sw.test.ts` mocks
  `self`/`caches`/`fetch`). For Glyde: cache the immutable shell for instant cold loads on the
  LAN; let Digest reads pass through (or add a deliberate read-through cache later, consciously).
- **SW registration in `+layout.svelte`**, guarded `browser && !dev && 'serviceWorker' in
  navigator`, failure logged not thrown (dev/SSR/HMR-safe).
- **`InstallHint.svelte`**: iOS Safari has no `beforeinstallprompt`, so a one-time, dismissible
  (localStorage-persisted) bottom strip tells the user to Share ‚Üí Add to Home Screen. Gated to
  real iOS Safari (UA sniff excluding CriOS/FxiOS/in-app webviews) and non-standalone. A
  `region` landmark with a ‚â•44px close button, never a modal over the content.
- **Viewport**: `interactive-widget=resizes-content` in `app.html` (keeps the layout sane when
  the iOS keyboard opens).
- **`crypto.randomUUID` fallback (LAN HTTP gotcha ‚Äî high value for Glyde).** `crypto.randomUUID`
  is **secure-context-only** and is `undefined` over plain HTTP on a LAN IP ‚Äî exactly the phone
  deployment ‚Äî so calling it throws and kills every mutation. Assay falls back to
  `crypto.getRandomValues` and assembles a v4 UUID by hand (`randomId()` in
  `labelling.svelte.ts`). If Glyde mints any client-side id over LAN HTTP, copy this.
- **iOS scroll/gesture CSS**: `overscroll-behavior-y: contain` / `overscroll-y-contain` on the
  card (kills Safari pull-to-refresh during a down-swipe); `touch-action: pan-y` /
  `[touch-action:pan-y]` on scrollable evidence so vertical reading scroll works while horizontal
  is the gesture; `overflow-x-clip` on the container so a drag transform can't create a sideways
  scrollbar; **‚â•44px touch targets** (`min-h-11`) on every button.

## 11. Accessibility patterns (reader-relevant)

- **`aria-live="polite"` announcer**: a single `sr-only` region whose text is a `$state` string
  (`loop.announcement`); every state change writes a sentence ("Passed. 12 remaining."). RSVP
  analogue: announce play/pause, position, speed changes for a screen-reader user.
- **Keyboard scope = focus containment** (WCAG 2.1.4): shortcuts fire only while the labelling
  surface holds focus (`surface?.contains(document.activeElement)`), never while typing in an
  input (`isEditable()` guard), and `event.preventDefault()` on every handled key (arrows must
  not scroll; a letter must not type itself).
- **Roving focus on advance**: an `$effect` re-focuses the stable card region when the item swaps
  so a keyboard user never drops to `<body>`.
- **Every gesture has a button equivalent** (WCAG 2.5.1): swipe pass/fail are mirrored by
  `VerdictBar` buttons carrying `aria-keyshortcuts`. Glyde: tap/swipe seek must have visible
  controls too.
- `:focus-visible` rings (`focus-visible:ring-2 focus-visible:ring-ring`), never blanket
  `outline:none`. Zero compiler `a11y_*` warnings is CI-fatal.

## 12. Testing & gates (lift selectively)

- **Vitest two-project split** (`vitest.config.ts`): a node `unit` project (the `load`-function
  and pure-logic tests) + a jsdom `component` project (`*.component.test.ts`, Testing Library),
  `resolve.conditions: ['browser']` in both. `npm run test` runs both. Glyde's CLAUDE.md already
