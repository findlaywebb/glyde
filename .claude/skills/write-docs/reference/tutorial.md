# Tutorial — learning-oriented

A tutorial takes a learner by the hand through a **guaranteed-to-succeed first experience**. The reader does not yet know what they need to know; you are the teacher and you are responsible for their success. (Diátaxis: practice + acquisition.)

## The discipline

- **The learner must succeed.** Every step works, in order, on a clean setup. If a step can fail, you've failed — pre-empt it or remove it.
- **Show, don't explain.** Concrete actions and visible results, not concepts. The learner builds confidence by *doing* and seeing it work. Save the "why" for the explanation doc and link to it.
- **Minimise choices.** No "you could also…", no configuration menus, no alternative paths. One road, paved.
- **Concrete over abstract.** Real commands, real output, real file names — not placeholders the learner has to resolve.
- **Repetition is fine.** A learner benefits from seeing a pattern several times; a reference reader would find it tedious. Different mode, different rule.
- **Point to what they'll see.** "You should now see `Server running on :3000`." Naming the expected result lets them self-correct.

## Shape

1. A one-line promise: what they'll have built by the end.
2. Prerequisites: the minimal, exact starting state (versions, install commands).
3. Numbered steps, each: do this → see this.
4. A checkpoint of success ("you now have a working X").
5. Where to go next (link to relevant how-tos / explanation).

## Don'ts

- Don't explain design decisions mid-flow — it breaks momentum and the guarantee.
- Don't offer options or cover edge cases — that's a how-to.
- Don't assume prior competence — that's also a how-to.
- Don't leave a step that "usually works". Usually isn't guaranteed.
